# 架构分层重构

本需求的目标是对此项目的架构分层，基于[docs/技术/PhoneClaw项目分层架构设计.md](../技术/PhoneClaw项目分层架构设计.md)进行重构，旨在：

1. 模块边界清晰：每个模块仅处理自己关心的事情，有助于代码的可维护性
2. 架构分层清晰：基于清晰的五层架构实现，并且按照统一规则分包，每一个组件所处的层级结构清晰可见
3. 数据流向清晰：基于层次规划数据流向，代码更加易于维护、推理

# 实现细节
## 通用实现细节
要求对于所有未在下文中显式提到的功能模块都按照 `PhoneClaw项目分层架构设计.md` 文档去重新进行分层设计。你可能需要将原有的某些类拆分或合并，组合成符合规范要求的新组件。

## Emu模块细节

- 将之前的`EmuApi`改成`EmuFacade`
- 将之前并不清晰的DomainService，改成`ScreenReader` 和 `ScreenOperator` 两种职责清晰的组件
- 将无障碍服务划分到`app`顶级模块中，不在`emu`包下

## LLM模块实现细节

- 完全拆分Data层Entity和Domain层对象，为模型提供商、模型都提供Domain层对象
- Skill的Repository拆分成多源的，分为只读的基于assets实现的内建源；和读写的基于文件实现的用户源。
- SkillEntity对象没有源的概念（内建/用户），但经过Domain层封装后，有了源的概念
- Skill的Repository提供内聚的缓存索引概念，用于快速检索以及获取元数据，当然这不会向外部暴露。Repository负责维护索引 -> 实际skill的一致性，并且仅用于索引作为加速目的。
  - assets使用一个单独的索引文件`assets/skills/index.json`来维护索引，由于是只读，对应Repository不需要维护一致性问题
  - user使用sqlite db来维护skill的元数据
  - 元数据不包含content，仅包含用于快速列表查询所包含的元数据信息，以及到真正skill源的ref
- 提供Session / Message概念。Data层使用数据库存储。Domain层使用Session暴露所有与当前会话、当前会话消息相关的增删改查服务。
- Integration层用于处理LangChain4j接入，它将提供 LangChain Agent -> PhoneClaw Agent；业务Model -> LangChain Model等整合能力
