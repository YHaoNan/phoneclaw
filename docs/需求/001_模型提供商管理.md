# 模型提供商管理

本需求文档用于描述PhoneClaw模型提供商管理模块的实现细节。

## 1. 概念定义

| 概念    | 说明                                                                                                                                                               |
|:-----:|:----------------------------------------------------------------------------------------------------------------------------------------------------------------:|
| 模型提供商 | 模型提供商定义了模型的提供厂商，模型提供商提供一系列模型。提供商中除了定义名字，最重要的还是定义提供商的连接方式，比如ApiKey等。<br/>我们打算加入的提供商有<br/><br/>OpenAI、Anthropic、DashScope、ZhiPu、Minimax、VLLM、Ollama等，目前我们仅提供OpenAI |
| 模型    | 基本上是一个数据类，它被挂到一个提供商下，具有模型名称、别名、模型能力定义等字段。目前我们最关心的能力就是是否支持视觉。                                                                                                     |



## 2. 功能梳理

### 2.1. 提供商定义

支持定义提供商，提供商定义分三个步骤：

1. 填写基本信息
   
   1. 填写提供商名称
   
   2. 选择提供商类型（目前仅提供OpenAI）

2. 填写特定提供商类型的参数信息（具体提供商的参数见下文）

3. 模型定义：为提供商添加模型
   
   1. 模型id填写
   
   2. 模型显示名称填写
   
   3. 模型是否具有视觉能力选择



### 2.2. OpenAI提供商

提供商名称：OpenAI



- apikey：apikey（非必填）

- base url: 基础url（必填，默认openai官方）

- chat completions url：聊天url（必填，默认openai官方 /v1/chat/completions）

- model url：模型获取url（必填，默认openai官方 /v1/models）

- 是否启用response api：（必填，默认不启用）

- response url：response url（必填，默认openai官方路径）

- 连接超时时间：必填，默认5s（单位秒）

- 请求超时时间：必填，默认60s（单位秒）



# 3. 实现细节

## 3.1. 按步骤分页

应该使用三个页面来实现模型管理功能

- 基础信息填写页 1/3

- 特定模型提供商配置填写页 2/3（每一个提供商的页面是定制的）

- 添加模型页面 3/3



关键点：

- 新增时，至少1,2页必须填完（填完第二页保存数据库）

- 支持编辑提供商，可以修改配置，但是不应该影响已有配置。绝不能出现修改完基础信息后，模型特定信息或挂在该提供商下的模型丢失的问题！！！绝不允许！！

- 涉密相关字段应采用加密编辑框，可以点击显示。保存后再次进入编辑也要加载到表单中，不能让用户每次都重新填写。

- 模型提供商管理列表中，每个提供商是一张卡片，默认是收起状态，仅显示提供商相关基础信息（仅包含提供商名称），展开后显示下面所有模型信息（仅包含显示名称）。并且展开后支持快速新增按钮直接进入添加模型页面（步骤3）



### 3.2. 部分api的模型快速添加

部分API支持获取模型列表功能，这些接口一般会返回提供商支持的模型列表，以及模型对应的能力。

若提供商支持，请在填写完步骤2后弹出弹窗：`当前提供商支持自动获取模型列表，是否自动获取` ，用户可选自动获取和跳过

若自动获取，则执行提供商的获取模型列表逻辑，并自动创建模型。



# 4. 技术细节

抽象组件：

- ModelProvider：代表模型提供商对象，包含提供商的私有配置和一些公共方法（抽象类或接口）
  
  - fromProviderConfig：基于数据库中存储的providerConfig创建ModelProvider对象
  
  - parseToProviderConfig：基于ModelProvider对象返回config信息
  
  - supportAutoFetchModelList(): Boolean：是否支持自动获取模型列表
  
  - fetchModelList(): List<Model>：获取模型列表

- Model：代表模型实体，数据类



数据库实体对象：

- ModelProviderEntity：包含基础配置，并使用modelProviderConfig字段保存模型私有配置

- ModelEntity：模型配置类，挂到ProviderEntity上



统一门面层(ModelProviderFacade)：

- 提供统一的业务逻辑门面层，让ui层不直接操作Repository，暴露：
  
  - 提供商增删改查
  
  - 模型增删改查
  
  - 获取所有模型列表（返回提供商列表->模型列表的二级嵌套结构，将在未来聊天界面中使用）



ui层：

- 每一个Step应该是一个页面或fragment，可以上一步下一步操作

- 第二个fragment依赖具体的ModelProvider实现配置项解析、配置项保存、模型自动获取（或者后续添加的其它能力）
