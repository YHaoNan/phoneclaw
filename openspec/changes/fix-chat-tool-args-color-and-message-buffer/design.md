## Context

当前聊天层存在三类可观测性问题：
1) 工具调用参数在 UI 卡片中缺失且未落库；
2) 悬浮窗的调用类型与执行结果颜色混淆，用户无法快速判断状态；
3) AI 流式消息缓冲未在消息边界清理，导致后续消息追加历史内容。

本次改动跨越事件模型、渲染状态机与持久化映射，属于跨模块行为一致性修复，需要先统一约束再进入实现。

## Goals / Non-Goals

**Goals:**
- 为工具/技能调用建立一致的事件结构，参数可展示、可持久化、可回放。
- 为悬浮窗建立确定性的颜色映射，按调用类型与成功/失败区分视觉状态。
- 修复流式消息缓冲生命周期，确保消息间隔离与结束清理。
- 通过自动化测试覆盖关键状态流转，防止回归。

**Non-Goals:**
- 不重构整体聊天页面架构。
- 不改动模型推理逻辑与工具执行顺序。
- 不引入新的主题系统，仅补齐状态色映射与消费逻辑。
- 不修改与本问题无关的模块（如模型提供商管理、非聊天业务页面与无关基础设施）。

## Decisions

### 1) 统一事件负载结构
- 决策：在聊天 UI 消费层统一使用 `CallEvent` 结构：`callType(tool|skill)`、`name`、`arguments`、`status(started|succeeded|failed)`、`result`、`error`、`timestamp`。
- 原因：避免工具与技能分别维护不同字段，减少序列化/渲染分支。
- 备选方案：保留现有分散字段并在 UI 端拼装；被拒绝，因为会持续引入字段遗漏（本次参数丢失即由此导致）。

### 2) 参数展示与持久化策略
- 决策：参数字段原样保留 JSON 字符串，并在 UI 卡片展示时做长度安全截断（仅展示层截断，不影响存储）。
- 原因：保证排障信息完整，同时避免超长文本破坏布局。
- 备选方案：入库前截断；被拒绝，因为会丢失诊断关键上下文。

### 3) 悬浮窗颜色状态矩阵
- 决策：建立固定映射：
  - tool + started
  - tool + succeeded
  - tool + failed
  - skill + started
  - skill + succeeded
  - skill + failed
- 原因：状态颜色由“类型 + 结果”双维度决定，消除“多状态同色”歧义。
- 备选方案：只按成功/失败区分两色；被拒绝，因为无法区分 tool 与 skill 的来源。

### 4) 流式消息缓冲生命周期
- 决策：每条 AI 消息创建独立 buffer（按 messageId 绑定）；收到完成事件后立即 flush 并释放；新消息开始必须重新初始化。
- 原因：解决 A/B 消息串扰，杜绝“最后一条等于历史总和”。
- 备选方案：全局单 buffer + reset 标记；被拒绝，因为并发/异常路径下容易漏 reset。

## Risks / Trade-offs

- [风险] 事件结构调整可能影响旧记录兼容 → [缓解] 对缺失字段提供默认值并兼容旧格式反序列化。
- [风险] 新增颜色 token 可能与现有主题冲突 → [缓解] 仅新增状态色变量，不改基础主题变量。
- [风险] Buffer 生命周期修复可能影响打字机动画节奏 → [缓解] 增加流式增量与完成事件联动测试，验证渲染节拍。
- [风险] 修复过程中误改无关模块引发连带回归 → [缓解] 以聊天链路为边界提交最小变更，并通过回归测试验证未触达模块行为不变。

## Migration Plan

1. 引入统一事件结构与映射适配层（兼容旧数据）。
2. 更新悬浮窗状态颜色映射并接入调用类型。
3. 重构 AI 消息 buffer 生命周期（创建、增量、完成、清理）。
4. 增加回归测试并在本地回放典型会话数据验证。
5. 发布后观察日志中的事件解析失败率与 UI 错误率；必要时可回滚到旧映射分支。
